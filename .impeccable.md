# Design Context for Photobooth IPH

## Users

**Primary Users**: Event photographers and production companies (e.g., Intania Production House) who operate photobooths at weddings, corporate events, parties, and other gatherings.

**Context**: Fast-paced event environments where speed and reliability are critical. Photographers are managing multiple workflows simultaneously—capturing photos, creating collages, uploading to Google Drive, and generating QR codes for guests. The application runs on Windows/macOS desktop computers connected to cameras and displays.

**Job to Be Done**: Efficiently capture, process, and share event photos with minimal friction. The interface must enable quick adjustments (collage layouts, frames, backgrounds) while maintaining professional quality.

**Emotional Goals**: Users should feel **confident & in control** and **efficient & productive**. The interface should reduce stress during events through clear workflows and predictable behavior.

---

## Brand Personality

**Voice**: Professional, capable, unobtrusive. The interface speaks with quiet competence—it doesn't demand attention but delivers reliably when needed.

**Three Words**: **Modern, Sleek, Minimal**

**Emotional Goals**:
- **Confidence**: Users trust the app to work flawlessly during live events
- **Efficiency**: Workflows are optimized for speed with minimal clicks
- **Calm**: The interface reduces cognitive load, not adds to it

**Anti-Personality**: The app is **not** playful, whimsical, or experimental. It's not trying to be "fun" or "creative" in the interface itself—it's a professional tool that enables creative work.

---

## Aesthetic Direction

**Visual Tone**: Dark, technical, precise—inspired by professional photography software like **Capture One**. The interface should feel like a precision instrument, not a consumer toy.

**References**:
- **Capture One**: Professional photo editing interface—dark backgrounds, subtle borders, clear visual hierarchy, technical but approachable
- **Adobe Lightroom/Camera Raw**: Similar professional photography aesthetic—organized, powerful, controlled

**Anti-References**:
- **Consumer social apps**: Avoid playful, rounded, "friendly" design patterns (e.g., Instagram, TikTok aesthetics)
- **Generic SaaS dashboards**: Avoid template-like card grids, generic gradients, or AI-generated design tropes
- **Overly minimal**: Don't strip away so much that the interface becomes cryptic or requires memorization

**Theme**: Dark mode only (currently #1e1e1e primary background). The app is used in dim event environments and dark interfaces are standard for professional photography software.

---

## Design Principles

These principles guide all design decisions for Photobooth IPH:

### 1. **Efficiency Over Elegance**
Speed matters more than aesthetics during live events. Prioritize:
- Fewer clicks to accomplish tasks
- Keyboard shortcuts for power users
- Predictable interaction patterns
- Clear visual hierarchy that reduces scan time

*Example*: A zone list item should show the most important info (zone number) prominently, not bury it among secondary details.

### 2. **Technical Clarity Over Simplification**
Users are professional photographers who understand technical concepts. Don't dumb down the interface:
- Show dimensions, file sizes, and technical details when relevant
- Use precise terminology (e.g., "X/Y position" not "where to put it")
- Honor user intelligence—don't hide complexity, organize it

*Example*: Properties panels can show numeric values directly, not abstracted sliders without readouts.

### 3. **Spatial Stability**
The interface should feel stable, not jumpy. Avoid:
- Accordions that push content down when opened (list reflow)
- Dynamic layouts that shift unexpectedly
- Animations that call attention to themselves

*Example*: Use fixed panels or overlays for properties editing rather than expanding list items that break spatial context.

### 4. **Professional Minimalism**
Clean and uncluttered, but not sparse or cryptic:
- Remove decorative elements that don't serve a function
- Use whitespace intentionally to group related elements
- Maintain sufficient contrast for readability in varied lighting
- Avoid visual noise (gradients, glows, unnecessary borders)

*Example*: Status indicators should be subtle (color tints, small badges) not attention-grabbing (glowing, animated).

### 5. **Confidence Through Feedback**
Users should never wonder "did that work?" or "what's happening?":
- Clear hover/focus/active states on all interactive elements
- Loading states for operations that take >200ms
- Success/error messages that explain what happened and what's next
- Visual confirmation of state changes (selected, locked, saved)

*Example*: Drag-and-drop should show clear drop zones with visual feedback (dashed borders, color changes).

## Technical Constraints

**Platform**: Desktop application (Tauri) running on Windows, macOS, and Linux
**Screen Size**: Default 1920x1080, should scale to 1366x768 minimum
**Input**: Mouse/keyboard primary, touch support secondary
**Performance**: Should feel responsive on mid-range hardware (animations ≤0.2s preferred)

**Accessibility**:
- WCAG 2.1 Level AA compliance for color contrast
- Keyboard navigation for all functionality
- Screen reader support for critical workflows
- Support for reduced motion preferences

## Design System Foundation

The application uses a well-established design token system defined in `src/styles/tokens.css`:

**Colors**:
- Primary background: `#1e1e1e` (dark gray)
- Accent blue: `#0078d4` (professional, not neon)
- Semantic colors for success, error, warning, info

**Typography**:
- System font stack: `-apple-system, BlinkMacSystemFont, 'Segoe UI', 'Roboto', 'Helvetica Neue', Arial, sans-serif`
- Monospace for technical values: `'SF Mono', 'Monaco', 'Consolas', 'Courier New', monospace`
- Size range: 10px–24px with careful hierarchy

**Spacing**:
- 4px scale: 4, 8, 12, 16, 20, 24, 32, 40px
- Used consistently for padding, margins, gaps

**Components**:
- Buttons: `.btn-primary`, `.btn-secondary`, `.btn-copy`
- Modals: Consistent structure with overlay, content, actions
- Forms: Input fields with consistent borders, focus states

All new designs should build on this existing system rather than creating new patterns.
